# 深度解释结构

## 深度层级

### 快速理解

适合“先告诉我它是什么”：

1. 一句话定义。
2. 它解决的具体问题。
3. 最关键的一步机制。
4. 一个使用边界。

### 标准解释

适合默认问答：

1. 定义和中心问题。
2. 旧方法卡在哪里。
3. 机制按因果顺序运行一遍。
4. 用具体输入和输出举例。
5. 与相邻方法比较。
6. 局限、成本和常见误解。

### 技术深挖

适合论文、架构或实现讨论。在标准解释基础上按需补充：

- 前提假设和适用条件。
- 组件、数据流或算法步骤。
- 公式中每一项代表什么，以及它改变了哪种行为。
- 原文证据、实验口径和对照基线。
- 失败模式、工程代价和仍未回答的问题。

不要把“更长”当成“更深”。每增加一层，都应回答读者当前问题中的一个真实缺口。

## 主张边界

### 原文主张

仅包括来源明确声称、定义、证明或实验直接报告的内容。保留必要限定词、数据口径和条件。

### 解释

把原文换成更易懂的说法，不新增结论。若为便于理解省略了细节，要指出省略会在哪些情况下失真。

### 推断

基于原文做出的影响判断、延伸应用或因果猜测。使用“我的推断”“这可能意味着”等标记，并说明依据与反例。

若来源没有表达某个结论，不得用“论文证明了”“作者认为”替自己的推断背书。

## 好解释的标准

- 读者能复述核心机制，而不只是记住类比。
- 读者知道它与旧方法或相邻概念的差别。
- 读者知道证据支持到哪里、推断从哪里开始。
- 读者知道什么时候不该使用它。
