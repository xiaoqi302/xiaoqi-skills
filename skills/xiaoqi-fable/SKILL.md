# 小七寓言配图

## 核心定位

为中文文章设计和生成 16:9 横版正文配图。目标不是商业插画、PPT 信息图、科技海报或可爱卡通，而是把文章里的关键判断、结构变化、时代隐喻、风险机会和个人选择，变成一张安静、清醒、有寓言感的黑白手绘图。

默认视觉 IP 是“小白”：白色空脸、深色短斗篷或长外套、细手脚、冷静观察者气质。小白必须参与画面的核心动作，不能只是站在旁边当装饰。

## 路由边界

使用本 skill 处理“文章观点 -> 寓言画面”的任务。若用户要技术机制白板图，优先用 `xiaoqi-tech-note`；若用户要中文白底知识卡，优先用 `xiaoqi-white`；若用户要米白底技术信息卡，优先用 `xiaoqi-info-card`；若用户要整篇公众号改写和配图编排，优先用 `xiaoqi-wechat`。

## 先读这些参考

按任务需要读取，不要一次塞满上下文：

- `references/style-dna.md`：未来寓言风、颜色、文字、禁忌。
- `references/xiaobai-ip.md`：小白 IP 的形象、性格、动作库和禁忌。
- `references/composition-patterns.md`：结构类型、原创隐喻方法和反复刻规则。
- `references/prompt-template.md`：单张生图提示词模板。
- `references/qa-checklist.md`：生成后检查和迭代规则。
- `examples/prompts.md`：只作低频提示词校准，不照抄构图。

## 工作流

### 1. 消化正文

先读用户给的正文、链接、Markdown 文件、Notion 页面或截图内容。提炼：

- 核心观点是什么
- 哪些段落承担认知转折
- 哪些内容适合被转成寓言式画面
- 哪些地方只适合文字，不需要图

不要平均配图。优先选择“认知锚点”，例如：时代门槛、座位变化、长期曲线、错误补救、风险交换、旧地图失效、系统校准、从旁观到入场。

### 2. 先出配图策略

如果用户只是说“分析怎么配图 / 思考哪些地方需要配图”，先给 shot list。每张图写清楚：

- 放在哪个段落后
- 图的主题
- 核心意思
- 结构类型
- 小白在图里做什么
- 建议元素
- 建议中文标注词

默认 4-8 张。文章很短时 1-3 张；长文也不要轻易超过 9 张。够用就好，避免把正文做成画册。

### 3. 单张生成

如果用户明确要求“生成 / 输出 / 做图 / 帮我生成”，不要停下来等确认；用内置 `image_gen` 每张单独生成。不要把多张图拼在一张里。

每张图只讲一个核心结构。提示词必须包含：

- 16:9 横版中文正文配图，或用户指定比例
- 纯白或近白背景
- 黑色手绘线稿，略带木刻寓言感
- 少量红色和蓝色中文手写批注
- 大量留白
- 小白作为核心动作主体
- 禁止 PPT、商业插画、科技 UI、幼稚可爱、复杂架构、左上角类型标题

不要复刻过往案例。案例只提供风格密度和小白参与方式，不能直接复用“项目交接 / 时间线山路 / 多跑者光谱 / 旧图书馆到田野 / AGI 门槛 / 牌桌换座”等已有构图，除非用户明确要求复刻某张图。每次都要从当前文章重新发明一个安静但成立的寓言隐喻。

### 4. 检查与迭代

生成后检查 `references/qa-checklist.md`。如果出现以下问题，优先重生成或局部编辑：

- 小白只是装饰
- 画面太满
- 太像流程图、PPT 或科技海报
- 中文太多或错字严重
- 左上角出现“流程图 / 系统架构 / 趋势分析”等类型标题
- 画风太可爱、商业、国风装饰化或科幻大片
- 背景不是干净白底或近白底

### 5. 保存交付

如果用户在 workspace 内工作，把最终图复制到：

```text
assets/<article-slug>-fable/
```

按顺序命名：

```text
01-topic-name.png
02-topic-name.png
```

保留原始生成文件，不要覆盖已有资产，除非用户明确要求替换。

## 输出口径

生成前的策略输出要短而准。生成后的交付要包含：

- 生成了几张
- 每张图的用途
- 保存路径
- 哪些图最稳，哪些图是可选

不要长篇解释风格理论；让图自己说话。
