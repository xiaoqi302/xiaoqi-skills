# 小七视觉质检

只对真实成图做视觉验收。提示词、文案和生成参数可以作为目标规格，但不能代替像素证据。

详细判定标准见 [references/qa-rubric.md](references/qa-rubric.md)，精确修复方法见 [references/repair-playbook.md](references/repair-playbook.md)。

## 输入

优先读取现有上下文，不重复追问：

- 必需：可实际查看的成图，包括附件、可访问链接或本地路径。
- 可选：目标比例或尺寸、发布平台及安全区、必须准确的文字、目标风格、参考图。
- 有参考图时先查看参考图，再比较风格或角色；没有参考图时只按已给风格规格检查，不声称完成参考图级一致性验证。

如果无法查看真实图片，停止审查，只问一个必要问题：`请附上图片，或提供一个我能访问的成图路径/链接。`

## 工作流

1. 查看真实图片，记录像素尺寸、可见裁剪、主要文字和画面区域。不要只读文件名、提示词或用户描述。
2. 整理目标规格。缺少可选规格时继续检查可观察问题，并把未知项标为“未提供”，不要连环追问。
3. 按 [references/qa-rubric.md](references/qa-rubric.md) 检查比例与裁剪、文字、层级、留白密度、风格、技术关系、IP 角色及伪 Logo/水印；不适用项标为 `N/A`。
4. 给出唯一结论：
   - `PASS`：没有阻塞交付的问题。
   - `REPAIR`：主体结构成立，问题可通过局部编辑修复。
   - `REGENERATE`：核心构图、文字、关系或风格大面积失效，局部编辑不可靠。
5. 按 `P0`、`P1`、`P2` 列问题。每项必须同时包含图中位置与可见证据、影响和精确局部修复指令；禁止只写“优化排版”“提高质感”等空泛建议。
6. 遵守动作边界：
   - 用户只要评审：只报告，不修改图片。
   - 用户明确要求修图且有可用工具：保留原图，优先局部编辑；每轮后重新查看真实结果并复检，最多 `2` 轮。
   - 用户要求修图但没有可用工具：只交付修复指令，明确说明未实际修改。
7. 只有看到修复后的真实图片并复检通过，才能把结论改为 `PASS`。只有提示词或修复计划时，不能声称视觉已通过。

## 输出格式

```text
结论：PASS / REPAIR / REGENERATE
目标规格：比例｜平台｜必须准确文字｜目标风格｜参考图状态

问题：
- [P0/P1/P2] 问题名称
  图中证据：具体区域 + 实际看到的现象
  影响：为什么阻塞或削弱交付
  修复指令：区域 + 保留项 + 精确改动 + 禁止改动项

通过项：列出已实际检查且通过的项目
未验证项：列出因规格缺失或不可见而无法判断的项目
修图执行：未执行 / 第 1 轮 / 第 2 轮；复检结论
```

没有问题时也要简要写明实际检查过的关键项目，不要只给一个 `PASS`。
